# GLC Audit Pipeline — Концепция и принятые решения

> Мастер-документ проекта. Фиксирует идею, все ключевые решения,
> принципы проектирования и границы v1 / v2.
> Техническая архитектура реализации — [ARCHITECTURE.md](./ARCHITECTURE.md) и синхронизированное обновление [ARCHITECTURE-upd.md](./ARCHITECTURE-upd.md) (TS-first v1, Supabase, go/no-go Python).

## 1. Суть идеи

GLC строит **полуавтоматический аудит-пайплайн**, который позволяет
провести полноценный бизнес- и технологический аудит любой компании,
обратившейся за консалтингом.

**Проблема:** Каждый аудит сейчас делается вручную — это занимает дни,
результат зависит от того, сколько времени есть, и не масштабируется.

**Решение:** Цепочка AI-агентов, каждый из которых отвечает за свою область
(техника, SEO, безопасность, маркетинг и т.д.). Агенты собирают данные,
анализируют, выставляют оценки. Alena ревьюит между фазами, дополняет
контекстом из интервью с клиентом, и финальный агент синтезирует всё
в структурированный отчёт с roadmap.

**Ценность для клиента:**
- Получает не сухой чеклист, а контекстные рекомендации с пониманием
  бизнес-реалий ("ПО устарело → вот почему миграция дорога → вот что
  можно сделать и за сколько").
- Видит что уже хорошо (не только проблемы).
- Получает quick wins — действия, которые можно внедрить завтра
  и сразу увидеть результат.
- Scorecard с числовой оценкой по каждой области — наглядно и эмоционально.

**Ценность для GLC:**
- Аудит любой компании за часы, а не за дни.
- Повторяемый, масштабируемый процесс.
- Фреймворк, который можно продавать как продукт.
- Автоматическая генерация вопросов для интервью под конкретную отрасль и стек.

## 2. Целевой рынок (v1)

Типичные компании **Майорки и Балеарских островов**:
- Туризм и гостеприимство (отели, апартаменты, туры, активности)
- Недвижимость (продажа, аренда, property management)
- Услуги (рестораны, клиники, юридические фирмы, gestorías)
- Малый и средний ритейл
- Морская индустрия (яхт-чартер, марины, сервис)

**Особенности рынка:**
- Сезонность (лето — хаос, зима — тишина)
- Мультиязычность (ES / EN / CA / DE)
- Зависимость от онлайн-каналов (Booking, Airbnb, TripAdvisor)
- Gestoría-культура (WhatsApp как бизнес-инструмент)
- Verifactu и обязательное e-invoicing — горячая тема
- Многие компании технически запущены, но готовы к изменениям

## 3. Принятые решения

### 3.1. Входной сигнал — сбор данных ДО запуска агентов

**Принцип:** Все данные от клиента собираются ПЕРЕД запуском
доменных агентов, а не между ними. Это критически важно для:
- Снижения галлюцинаций — агенты работают с реальными данными, а не с допущениями.
- Качества отчёта — каждый агент видит полную картину.
- Репутации GLC — мы не можем выдавать предположения за факты.

**Порядок:**

| Шаг | Что происходит | Кто делает | Результат |
|-----|----------------|------------|-----------|
| 1 | Клиент даёт URL + заполняет Intake Brief | Клиент + Alena | Запись брифа в **Supabase** (логически `intake_brief`) |
| 2 | Recon краулит сайт, собирает объективные данные | Агент (авто) | `audit_recon` + `collected_data` |
| 3 | Recon обогащает бриф: подставляет найденные факты, уточняет пробелы | Агент (авто) | Обновлённый бриф |
| 4 | Alena ревьюит: всё ли собрано? Нужны ли дополнительные вопросы? | Alena | ★ REVIEW POINT |
| 5 | Доменные агенты (параллельные крылья в v1) | Агенты (авто) | Строки в `audit_domains` |
| 6 | Alena ревьюит результаты всех агентов | Alena | ★ REVIEW POINT |
| 7 | Strategy Agent синтезирует roadmap | Агент (авто) | `audit_strategy` + отчёт (API/экспорт) |

**Ключевое отличие от предыдущей версии:**
- Раньше: Recon → агенты → бриф → аналитические агенты (данные собирались постепенно).
- Сейчас: Бриф + Recon → ВСЕ агенты сразу (данные собраны заранее).
- Если что-то НЕ известно, агент **явно маркирует это как unknown**,
  а не придумывает правдоподобный ответ.

### 3.2. Уровень автоматизации — полуавтоматический

- Агенты собирают и анализируют **самостоятельно**.
- Alena **ревьюит между фазами** и дополняет.
- Три точки ревью: после Recon, после всех доменов, после финального Roadmap.
- Клиент НЕ видит промежуточные данные — только финальный отчёт.

### 3.3. Формат отчёта — Markdown (v1), PDF/PPTX/HTML (v2)

Начинаем с Markdown, потому что:
- Быстро генерируется.
- Легко редактируется вручную.
- Конвертируется в PDF через pandoc.
- В v2 добавим презентацию (PPTX) и интерактивный HTML-отчёт.

### 3.4. Scorecard — числовой балл + цвет + текст

Каждый домен оценивается по шкале 1–5:

| Балл | Цвет | Статус | Значение |
|------|------|--------|----------|
| 5 | 🟢 | Excellent | Лучшие практики, минимум улучшений |
| 4 | 🟢 | Good | Работает хорошо, есть minor improvements |
| 3 | 🟡 | Needs Improvement | Функционирует, но есть значимые пробелы |
| 2 | 🟠 | Significant Issues | Серьёзные проблемы, влияют на бизнес |
| 1 | 🔴 | Critical | Требуют немедленных действий |

По каждому домену в отчёте: что хорошо ✅ → что плохо ⚠️ → quick wins 🚀 → рекомендации 💡.

Каждая рекомендация содержит: описание проблемы, почему она важна,
что конкретно сделать, сколько это займёт, сколько стоит,
и какой бизнес-эффект ожидать.

### 3.5. Стек — TypeScript-first (v1) + опциональный Python (v2+)

**v1 (текущая реализация):**

- **Node.js 20+ / TypeScript** — backend, оркестрация, агенты, API.
- **Express** — REST `/api/*`, `requireAuth`, интеграция с Supabase service role на сервере.
- **Supabase** — PostgreSQL, Auth, Realtime; state аудита в таблицах (`audits`, `audit_recon`, `audit_domains`, …), не единый файл на диске.
- **Claude API** (`claude-sonnet-4-20250514`) — один вызов на фазу через `BaseAgent`.
- **`cheerio` + `fetch`** — программный краулинг/парсинг (коллекторы без LLM).
- **Zod** — валидация структурированного вывода агентов.
- **React 18 + Vite** — UI портфолио, монитор пайплайна, workspace аудита.

**Целевые улучшения (не блокируют go-live):** вынести пороги в единый конфиг; вынести длинные промпты в `server/prompts/*.md`.

**Python (не часть v1):** отдельный **compute-микросервис** только после go/no-go: минимум **3 из 4** критериев в [ARCHITECTURE-upd.md](./ARCHITECTURE-upd.md) (подраздел **12.3**) — heavy crawl/OCR/NLP/ML, доля новой логики на Python, команда, измеримый проигрыш TS. Оркестрация остаётся на TS.

**n8n / внешняя оркестрация:** опционально в v2+; до отдельного ADR основной поток — Express `PipelineService`.

### 3.6. Сбор данных — всё собираем в первом шаге

**Ключевой принцип:** Данные собираются ДО запуска агентов, а не между ними.
Каждый агент — отдельный специалист, и каждый знает, что ему нужно.
Но клиент НЕ должен заполнять 6 разных анкет — он заполняет ОДИН Intake Brief,
который содержит вопросы от всех агентов.

**Как это работает:**

```
ШАГ 1: СБОР ДАННЫХ (до запуска любых агентов)
┌────────────────────────────────────────────────────────┐
│                                                        │
│  Клиент получает Intake Brief                         │
│  (один документ, ~35 вопросов, ~40 мин)               │
│                                                        │
│  Вопросы сгруппированы по секциям, но клиент           │
│  не знает, что за каждой секцией стоит отдельный       │
│  агент — для него это один цельный процесс.            │
│                                                        │
│  Для каждого вопроса:                                  │
│  🔴 Обязательный — без него аудит не запускается       │
│  🟡 Рекомендуемый — повышает качество                  │
│  🟢 Опциональный — дополнительная глубина              │
│                                                        │
│  Если клиент не ответил на 🟡/🟢 вопрос,              │
│  агент маркирует эту область как [UNKNOWN]             │
│  и НЕ делает допущений.                                │
│                                                        │
└────────────────────────────────┬───────────────────────┘
                                │
                                ▼
ШАГ 2: RECON (автоматический, параллельно с брифом)
┌────────────────────────────────────────────────────────┐
│  Краулит сайт, собирает объективные данные.            │
│  Обогащает бриф: подставляет найденные факты,          │
│  выявляет расхождения с ответами клиента.              │
└────────────────────────────────┬───────────────────────┘
                                │
                                ▼
★ REVIEW POINT: Alena проверяет полноту данных.
  Если критических пробелов нет → запуск всех агентов.
  Если есть → дособирает у клиента.
                                │
                                ▼
ШАГ 3: ВСЕ ДОМЕННЫЕ АГЕНТЫ (с полным набором данных)
```

**Каждый агент предоставляет свои входные требования:**

1. **Required inputs (🔴)** — без этого агент НЕ выдаёт оценку по этому пункту.
   Вместо этого пишет: *"Insufficient data to assess. Recommendation: [что нужно предоставить]"*.
2. **Recommended inputs (🟡)** — повышает качество. Без них агент работает,
   но маркирует gaps: *"Assessment based on external data only. Internal data would improve accuracy."*
3. **Nice-to-have (🟢)** — дополнительная глубина.
4. **Auto-collectable (✅)** — агент соберёт сам из Recon.

**Пример для Marketing Agent:**

| Приоритет | Вопрос / данные | Зачем нужно |
|-----------|-----------------|-------------|
| 🔴 Required | Кто ваш идеальный клиент? | Оценка соответствия сайта ЦА |
| 🔴 Required | Чем отличаетесь от конкурентов? | Оценка УТП |
| 🟡 Recommended | Какие каналы привлечения используете? | Анализ маркетинг-микса |
| 🟡 Recommended | Какой средний чек и LTV? | Контекст для ROI |
| 🟢 Nice-to-have | Есть ли brand guidelines? | Оценка consistency |
| ✅ Auto | Тексты сайта, структура офферов, CTA | Парсинг из Recon |

**Пример для Tech Infrastructure Agent:**

| Приоритет | Вопрос / данные | Зачем нужно |
|-----------|-----------------|-------------|
| ✅ Auto | Стек (CMS, фреймворки, CDN) | HTML-детект |
| ✅ Auto | Performance metrics | Lighthouse |
| 🟡 Recommended | Кто обслуживает сайт? | Контекст рекомендаций |
| 🟡 Recommended | Есть ли бэкапы, мониторинг? | Оценка зрелости |
| 🟢 Nice-to-have | Доступ к hosting panel / Git | Глубокий анализ |

### 3.7. Контекстная релевантность — не все рекомендации одинаково полезны

**Принцип:** Агент должен трезво оценивать, нужна ли вообще
рекомендация в конкретном контексте.

**Пример:** Крупный госпиталь, которому все на острове и так знают.
SEO-агент должен понять: у этого клиента органический трафик
обеспечен узнаваемостью бренда. Рекомендовать "улучшите мета-теги
для увеличения CTR" — бессмысленно. Зато "ваша страница записи
на приём не индексируется, и пациенты не находят её в Google" —
это ценная находка.

**Как это реализуется:**

Каждый агент получает из Recon:
- `industry` + `industry_subcategory` — отрасль
- `estimated_size` — размер компании
- `business_model` — как зарабатывают
- `competitive_landscape_notes` — контекст конкуренции
- `mallorca_relevance` — специфика рынка

На основе этого агент:
1. **Калибрует ожидания** — что считать "нормой" для этого типа бизнеса.
2. **Фильтрует рекомендации** — не предлагает enterprise-решения solo-бизнесу.
3. **Приоритизирует по бизнес-impact** — а не по "лучшим практикам" абстрактно.
4. **Предлагает варианты** — "вот 2-3 пути, вот trade-offs каждого".

**Шкала релевантности рекомендации:**

| Тег | Значение |
|-----|----------|
| `CRITICAL` | Без этого бизнес теряет деньги/клиентов прямо сейчас |
| `HIGH_VALUE` | Значительный эффект при разумных затратах |
| `NICE_TO_HAVE` | Улучшит, но не критично для этого типа бизнеса |
| `NOT_APPLICABLE` | Не релевантно для данной компании (с объяснением почему) |

### 3.8. Unified Intake Brief — единый бриф, ПЕРЕД всем остальным

Intake Brief — это **первое, что получает клиент** (или заполняется
совместно на discovery-звонке). Без минимального набора ответов (все 🔴)
аудит не запускается.

**Структура:**

```
INTAKE BRIEF: [Company Name]
Заполняется: клиентом или совместно на discovery-call

═══ О ВАШЕМ БИЗНЕСЕ (5 min) ═══
1. 🔴 Кратко опишите ваш бизнес своими словами
2. 🔴 Кто ваш идеальный клиент?
3. 🔴 Какая главная проблема, которую вы хотите решить?
4. 🔴 Чем вы отличаетесь от 2-3 ближайших конкурентов?

═══ МАРКЕТИНГ И ПРОДАЖИ (10 min) ═══
5. 🟡 Какие каналы привлечения работают лучше всего?
6. 🟡 Какой средний чек?
7. 🟡 Есть ли сезонность? Как влияет?
8. 🟢 Есть ли brand guidelines / фирменный стиль?
9. 🟢 Какой бюджет на маркетинг?

═══ ТЕХНОЛОГИИ И САЙТ (10 min) ═══
10. 🟡 Кто обслуживает ваш сайт?
11. 🟡 Какие программы/сервисы используете ежедневно?
12. 🟡 Есть ли бэкапы и мониторинг?
13. 🟡 Планируется ли редизайн или миграция?
14. 🟢 Есть ли Google Search Console / Analytics?

═══ ПРОЦЕССЫ И РУТИНА (10 min) ═══
15. 🔴 Опишите типичный рабочий день
16. 🔴 Какие задачи повторяются каждый день/неделю?
17. 🟡 Сколько сотрудников? Кто за что отвечает?
18. 🟡 Есть ли описанные бизнес-процессы?
19. 🟡 Какие данные передаёте вручную между системами?

═══ БЕЗОПАСНОСТЬ И COMPLIANCE (5 min) ═══
20. 🟡 Принимаете ли онлайн-платежи?
21. 🟡 Используете ли e-invoicing / Verifactu?
22. 🟢 Был ли аудит безопасности раньше?

═══ ОЖИДАНИЯ ОТ АУДИТА (5 min) ═══
23. 🔴 На какие области хотите сделать упор?
24. 🟡 Какой бюджет на улучшения (ориентировочно)?
25. 🟡 Какие сроки? Есть ли дедлайны?

Total: ~40-45 min
Required (🔴): 7 вопросов — без них аудит НЕ запускается
Recommended (🟡): 13 — сильно повышают качество
Nice-to-have (🟢): 5 — дополнительная глубина
```

**SLA по качеству брифа:**
- Все 🔴 вопросы отвечены → аудит запускается.
- 70%+ 🟡 вопросов отвечены → полноценный аудит.
- < 50% 🟡 → агенты работают, но маркируют gaps в отчёте.
- Неизвестные параметры → `[UNKNOWN]`, не допущение.

**Адаптация под отрасль:**
После Recon бриф может быть дополнен отраслевыми вопросами:
- Отель: "Cloudbeds — единственная система, или есть другие каналы?"
- Недвижимость: "На каких порталах размещаете объекты?"
- Ресторан: "Есть ли система бронирования столиков?"

Но базовый набор вопросов — универсальный и собирается ДО Recon.

### 3.9. Порядок разработки — начинаем с Recon

Recon Agent — логическое начало пайплайна. Без него остальные агенты
не имеют входных данных. Уже реализован и протестирован на glctech.es.

### 3.10. Anti-Hallucination Policy (КРИТИЧЕСКИ ВАЖНО)

**Репутация GLC — главный актив. Выдумывать данные строго запрещено.**

Это не рекомендация, а жёсткое правило, встроенное в промпты всех агентов
и в архитектуру пайплайна на уровне дизайна.

**Правила для каждого агента:**

**Правило 1: Только факты или явная маркировка.**
Каждое утверждение в отчёте может быть только одного из трёх типов:

| Тип | Маркировка | Пример |
|-----|------------|--------|
| Факт (проверенный) | `[VERIFIED]` | "SSL-сертификат истекает 2026-05-01" |
| Данные от клиента | `[FROM BRIEF]` | "Клиент использует Cloudbeds для бронирований" |
| Неизвестно | `[UNKNOWN]` | "Наличие бэкапов: данные не предоставлены" |

**`[ASSUMED]` — ЗАПРЕЩЁН.** Агент НЕ ИМЕЕТ ПРАВА делать допущения
и выдавать их за факты. Если данных нет — пишем `[UNKNOWN]`
и рекомендуем что нужно проверить.

**Правило 2: Evidence-based рекомендации.**
Каждая рекомендация привязана к конкретному источнику:

```json
{
  "recommendation": "Обновить WordPress с 5.8 до 6.5",
  "evidence": {
    "type": "auto_detected",
    "source": "HTML meta generator tag",
    "page": "https://example.com/",
    "raw_value": "<meta name='generator' content='WordPress 5.8'>",
    "confidence": "high"
  }
}
```

Если источника нет — рекомендация не попадает в отчёт.

**Правило 3: Confidence levels.**
Каждый факт и рекомендация имеет уровень уверенности:

| Level | Критерий присвоения | Как отображается в отчёте |
|-------|---------------------|--------------------------|
| `high` | Машинный скан / объективный факт, проверяемый повторным запуском | Показывается как факт |
| `medium` | Аналитика на основе ≥2 независимых сигналов | "Based on our analysis of..." |
| `low` | Один слабый сигнал ИЛИ противоречивые данные | НЕ попадает в отчёт без ревью Alena |

**Правило 4: Quality gate перед финальным отчётом.**
Отчёт НЕ генерируется, если:
- Есть рекомендации с `confidence: low` без пометки Alena.
- Есть прямые противоречия между данными разных агентов.
- Не все 🔴-вопросы из брифа получили ответ.

**Правило 5: Что делать, когда данных нет.**
Если агент не может оценить область из-за нехватки данных,
он НЕ ставит оценку, а пишет:

```
### ⚙️ Tech Infrastructure
### Score: —/5 ⚪ Insufficient Data

We were unable to fully assess this domain.
**Missing data:** [список того, что нужно предоставить]
**What we CAN say based on available data:** [только проверенные факты]
**Recommendation:** Provide access to [X] for a complete assessment.
```

Это честнее и профессиональнее, чем выдумать оценку 3/5.

**Правило 6: Агенты НЕ имеют права менять данные брифа.**
Если автоданные противоречат брифу — только `[DISCREPANCY]` и запрос
на уточнение. Агент не "исправляет" ответ клиента.

**Правило 7: Разделение внутренних и клиентских меток.**
В **внутреннем state** (строки `audit_domains` / логический JSON из [ARCHITECTURE-upd §3](./ARCHITECTURE-upd.md#3-data-contract-json)) хранятся полные метки: `[VERIFIED]`, `[FROM BRIEF]`,
`[UNKNOWN]`, `confidence`, `evidence_refs`.
В клиентском отчёте — мягкие формулировки:
- `[VERIFIED]` → показывается как факт (без метки)
- `[FROM BRIEF]` → "Based on information provided by the client"
- `[UNKNOWN]` → "Data not available — we recommend verifying..."
- `confidence: medium` → "Based on our analysis of..."

Внутренние теги остаются для ревью Alena, но не перегружают отчёт клиента.

### 3.11. Verification Layer — внутренний факт-чекинг

Перед тем как данные попадут в отчёт, каждый агент проходит
внутреннюю самопроверку:

**Шаг 1: Self-check (встроен в промпт агента).**
Агент перед записью результата отвечает на чеклист:
- "Привязана ли каждая рекомендация к конкретной найденной проблеме?"
- "Нет ли противоречий с данными из брифа?"
- "Не выдумал ли я цифры или факты?"
- "Для каждого утверждения: могу ли я указать источник?"

**Шаг 2: Cross-agent consistency check (автоматический).**
После работы всех агентов — автоматическая проверка:
- Если Tech Agent сказал "WordPress", а SEO Agent сказал "React SPA" — противоречие.
- Если Marketing Agent написал "нет УТП", а на сайте явно есть → проверить.

**Шаг 3: Human review (Alena).**
Alena видит все `confidence: medium/low` маркировки
и фокусируется на них, а не читает весь отчёт подряд.

**Шаг 4: Human QA Checklist (для ревьюера).**
Воспроизводимый чеклист — если ревьюером станет кто-то кроме Alena:

1. ☐ Все рекомендации привязаны к evidence? (нет "висящих в воздухе")
2. ☐ Нет `confidence: low` без явной пометки?
3. ☐ Нет `[DISCREPANCY]` без резолюции?
4. ☐ Scorecard оценки адекватны контексту? (не завышены/занижены для отрасли)
5. ☐ Quick wins реалистичны для этого размера бизнеса?
6. ☐ Tone of voice отчёта — профессиональный, не снисходительный?
7. ☐ Нет фактических ошибок в описании бизнеса клиента?

### 3.12. Provenance — происхождение каждой находки

В JSON домена (в БД — поле output соответствующей строки `audit_domains`) для каждой рекомендации хранится:

```json
{
  "item": "Нет Content-Security-Policy header",
  "impact": "high",
  "evidence_refs": [
    {
      "type": "http_header_scan",
      "url": "https://example.com",
      "finding": "CSP header absent in response headers",
      "timestamp": "2026-03-09T10:15:00Z"
    }
  ],
  "agent_name": "security_compliance",
  "confidence": "high",
  "data_source": "auto_detected",
  "verified_by": null
}
```

Это позволяет:
- Клиенту проверить: "откуда это взялось?"
- Alena быстро ревьюить: видно, где факт, а где интерпретация.
- Улучшать агентов: видно, где были ошибки.

### 3.13. Handling unknown parameters and partial failures

Когда клиент не предоставил данные (не ответил на 🟡/🟢 вопрос),
агент действует по строгому протоколу:

| Ситуация | Действие агента | Пример |
|----------|-----------------|--------|
| 🔴 вопрос без ответа | Аудит этого домена НЕ запускается | "Нет данных о целевой аудитории → Marketing Agent не может оценить УТП" |
| 🟡 вопрос без ответа | Агент работает с доступными данными, маркирует gap | "Бэкапы: [UNKNOWN]. Рекомендуем уточнить наличие бэкапов." |
| 🟢 вопрос без ответа | Агент работает нормально, просто без этого слоя | Нет brand guidelines → агент не оценивает brand consistency |
| Auto-данные противоречат брифу | Маркируется как `[DISCREPANCY]` для ревью Alena | "Бриф: 'используем WordPress'. Recon: React SPA. Требуется уточнение." |
| Auto-данные неоднозначны | `confidence: medium`, не утверждается как факт | "Возможно используется Magento (обнаружен паттерн, но не подтверждён)" |
| Агент упал (ошибка API/парсинга) | `status: "error"`, данные НЕ записываются | Strategy Agent строит roadmap без этого домена, помечает "Not assessed (technical error)" |

**Поведение Strategy Agent при частичных отказах:**
Если 1–2 домена в `status: "error"` или `"insufficient_data"`:
- Roadmap строится по доступным доменам.
- Executive Summary явно указывает: "Covers N of 6 domains. [Domain X] could not be assessed."
- Scorecard показывает `— ⚪` для этих доменов, overall score — только по доступным.
- Alena получает уведомление для решения: перезапустить агент или выпустить как есть.

### 3.14. Конфигурация — пороги и правила в `config.yaml`

Все числовые пороги и правила вынесены в конфигурационный файл,
а НЕ захардкожены в коде агентов. Это включает: лимиты краулинга,
SLA по брифу, критерии confidence, параметры quality gate, token budgets,
настройки логирования. Подробная структура — в `ARCHITECTURE.md` §6.4.

## 5. Архитектура пайплайна — Data First

Принципиальная особенность: **сначала данные, потом анализ**.

**Шаг 1: Data Collection (до агентов)**
Intake Brief от клиента + Recon (автокраулинг).
Результат: полный набор входных данных для всех агентов.
Alena ревьюит полноту. Если 🔴-пробелы — дособирает.

**Шаг 2: Автоматическое крыло (Phases 1–4)**
Tech Infrastructure, Security, SEO, UX — данные видны снаружи.
Агенты собирают 80%+ информации сами, бриф обогащает контекст.
Не зависят друг от друга, запускаются параллельно.

**Шаг 3: Аналитическое крыло (Phases 5–6)**
Marketing/УТП, Automation/Processes — зависят от брифа.
С полным брифом дают глубокий анализ.
Без полного брифа работают с доступными данными + маркируют `[UNKNOWN]`.

**Шаг 4: Синтез (Phase 7)**
Strategy & Roadmap — объединяет все данные.
Alena ревьюит финальный отчёт.

**Ключевое:** Оба крыла получают ВСЕ данные сразу.
Разница только в том, насколько каждое крыло зависит от брифа vs автоданных.

## 6. Принципы отчёта

### 6.1. Не сухой чеклист, а история

Отчёт должен читаться как рассказ: "вот кто вы, вот что мы нашли,
вот что работает хорошо, вот где проблемы, вот что делать".
Не список из 50 пунктов, а приоритизированный narrative.

### 6.2. Quick wins с эмоциональным эффектом

Каждый отчёт содержит 3–5 действий, которые клиент может
внедрить за неделю и увидеть результат. Это создаёт доверие
и мотивирует на дальнейшую работу.

Примеры quick wins по доменам:
- **Security:** добавить CSP header (1 час, €0) → защита от XSS
- **SEO:** добавить meta description на 5 страниц (2 часа) → CTR +15-30%
- **UX:** увеличить touch targets на мобильной версии (3 часа) → снижение bounce rate
- **Automation:** настроить auto-reply на email (30 мин) → клиенты не ждут

### 6.3. Контекстные рекомендации

Не "ваш WordPress устарел", а:
- Текущая версия: 5.8 (2021)
- Текущая стабильная: 6.5
- Риск: 12 известных уязвимостей
- Миграция займёт: 4–8 часов
- Стоимость: €200–500 (подрядчик)
- Альтернатива: оставить, но обновить security plugins (2 часа, €0)
- Бизнес-эффект: защита данных клиентов, соответствие GDPR

### 6.4. Что хорошо — тоже отмечаем

Каждый домен начинается с "✅ What's Working Well".
Это не формальность — клиент должен увидеть, что аудитор
заметил и оценил то, что уже сделано правильно.

Поиск уникальной суперсилы: что делает этот бизнес незаменимым?
Не УТП как маркетинговый термин, а глубокое понимание —
какая одна вещь, если её усилить, изменит всё.

### Расширенные форматы отчёта
- PDF с дизайном GLC (Ink Blue + Warm Terracotta)
- PPTX-презентация для встреч
- Интерактивный HTML-отчёт (кликабельная презентация)
- Автоматическая генерация тестового сайта-прототипа

## 8. Особенности и edge cases

### 8.1. Отраслевая адаптация
Пайплайн должен адаптироваться под отрасль клиента.
Пример: для отеля проверяем booking-интеграции и channel manager,
для агентства недвижимости — MLS и портальные интеграции,
для ресторана — системы бронирования столиков и отзывы.

### 8.2. Мультиязычность
Сайты на Майорке часто мультиязычные (ES/EN/CA/DE).
Агенты должны проверять hreflang, качество перевода,
и SEO-структуру для каждого языка.

### 8.3. Сезонность
Многие бизнесы Майорки сезонные. Агенты учитывают это:
"летом вам нужно X, зимой — Y, вот как подготовиться".

### 8.4. Gestoría-фактор
Многие компании работают через gestorías, которые отправляют
документы через WhatsApp. Это не баг, это реальность рынка.
Рекомендации должны быть реалистичными, а не "переходите на SAP".

### 8.5. Verifactu и e-invoicing
С 2026 в Испании обязательное электронное выставление счетов.
Каждый аудит должен проверять готовность к этому.

### 8.6. Слабые/пустые сайты (Foundational Mode)
Если сайт очень маленький (< 5 страниц) или на конструкторе с минимумом
доступа, аудит автоматически переключается в **Foundational Mode**:
фокус на структуру, контент и базовый стек, а не на продвинутую
автоматизацию/процессы. Recon маркирует `coverage: low` и агенты
адаптируют глубину.

## 9. Продуктовая лестница

### 9.0. Free UX Snapshot (бесплатный лид-магнит)

**Цель:** Клиент вводит URL → через 5-10 минут получает красивый
мини-отчёт по одному домену. Видит качество и глубину → хочет полный аудит.

**Почему UX:**
- Максимальный эмоциональный эффект ("вот почему люди уходят с вашего сайта").
- Почти всё собирается снаружи (HTML + Lighthouse), минимум токенов.
- Визуально впечатляет: скриншоты/описания проблемных мест.

**Что входит:**

| Показываем | НЕ показываем |
|------------|---------------|
| Scorecard UX (1-5 + цвет) | Остальные 5 доменов (locked) |
| 2-3 из 7+ найденных issues | Полный список issues |
| 2 quick wins (самые яркие) | Recommendations с оценкой ROI |
| Мини-сравнение с 1 конкурентом (2-3 метрики) | Полный competitor benchmark |
| Структура полного отчёта (видны заголовки секций) | Содержание других секций |

**Что видит клиент в конце:**
```
═══════════════════════════════════════════
You've seen 2 of 7 UX improvements found.
Your full UX audit is part of GLC Express Audit.

📊 Express Audit covers: Tech + Security + SEO + UX
📊 Full Audit adds: Marketing, Automation, Strategy & Roadmap
═══════════════════════════════════════════
```

**Ограничения (чтобы не съесть маржу):**
- 1 домен, до 5 страниц.
- Только авто-данные (без брифа).
- Strategy/Roadmap агент НЕ вызывается.
- Confidence/provenance — внутренние, не показываются клиенту.
- Оценки ROI/денег: только "высокий потенциал", без точных цифр.

**Вход:** URL + email.
**Выход:** Markdown/PDF на email + CTA на Express/Full.

### 9.1. Express Audit (платный, low ticket)

Автоматическое крыло: Phases 0–4 (Tech, Security, SEO, UX).
Лёгкий бриф (только 🔴 вопросы, ~15 мин).
Базовый Competitor Benchmark (1 конкурент).
Короткий roadmap (только quick wins + medium term).

Время: 2–3 часа. Результат: 5-7 страниц + scorecard + mini-roadmap.
Отчёт: **Owner One-pager** + общий scorecard.

### 9.2. Full Audit (core product, v1)

Все 8 фаз. Полный Intake Brief (~40-45 мин).
Competitor Benchmark (2-3 конкурента).
Полный roadmap: quick wins → medium → strategic.
Рекомендация пакета GLC.

Отчёт: **Full Report** + отдельные Owner / Marketing / Tech views.

### 9.3. Continuous Audit / Quarterly Check-up (v2)

Раз в квартал перезапуск автокрыльев + обновление брифа.
Показывает динамику vs предыдущий аудит.
Короткий отчёт: только изменения + новые рекомендации.
Создаёт подписочную модель.

### 9.4. Competitor Benchmark Audit (v2)

Лёгкий Recon на 2–3 конкурентов + сравнительная таблица.
Продаётся как отдельный продукт или add-on к Full Audit.

### 9.5. Пакетирование и связка с отчётами

| Пакет | Домены | Отчёты | Competitor | Для кого |
|-------|--------|--------|------------|----------|
| Free Snapshot | UX only | One-pager (частичный) | 1 мини-сравнение | Холодные лиды |
| Express | Tech, Security, SEO, UX | Owner One-pager + scorecard | 1 конкурент | Быстрая диагностика |
| Full Audit | Все 8 | Full + Owner + Marketing + Tech views | 2-3 конкурента | Core product |
| Audit + Sprint | Все 8 + внедрение 2-4 quick wins | Full + action tracker | 2-3 конкурента | Быстрые результаты |
| Continuous Ops | Авто-крыло quarterly + бриф update | Delta report (изменения) | Обновляемый | Retainer-клиенты |

### 9.6. Клиент включает/выключает домены

В Full Audit клиент может выбрать scope:
- "Web & Marketing only" → Phases 1, 3, 4, 5
- "Tech & Security only" → Phases 1, 2
- "Full Digital Operations" → все фазы

Невыбранные домены → "Available in extended audit" в отчёте.
Это НЕ снижает стоимость — меняет фокус и глубину.

### 9.7. Sales-артефакты поверх пайплайна

Из пайплайна генерируются материалы для продажи:
- **Sample Report** (обезличенный) — демонстрация качества.
- **2-3 слайда** для discovery-call: "Как работает пайплайн", "Что вы получаете через 72 часа", "До/После".
- **Free Snapshot** сам по себе — lead magnet и демо продукта.

### 9.8. Action Plan как экспортируемый артефакт

Roadmap генерируется не только как секция отчёта, но и как
отдельный файл `action_plan.csv`:

```
Title, Domain, Priority, Effort_Hours, Cost_EUR, Impact, Owner, Type
"Add CSP header", Security, High, 1, 0, "XSS protection", Client, Quick Win
"Update meta descriptions", SEO, High, 2, 0, "CTR +15-30%", Client, Quick Win
...
```

Импортируется в Trello / Asana / ClickUp / Notion.
Отчёт не "умирает" в PDF — становится рабочим инструментом.

## 10. Конкурентный анализ внутри пайплайна

### 10.1. Как встроен в flow
- Recon собирает 2–3 кандидатов-конкурентов (SERP, Google Business, TripAdvisor).
- Alena на ревью подтверждает или корректирует список.
- Запускается упрощённый Recon на конкурентов (структура, УТП, базовые тех-метрики).
- Данные попадают в `competitors[]` в логическом state (recon / JSON-поля аудита в Supabase).

### 10.2. Критерии выбора конкурентов

| Слот | Тип | Зачем |
|------|-----|-------|
| Competitor A | Прямой локальный конкурент | Реальное сравнение на рынке |
| Competitor B | "Идеальный" (лучший в нише) | Бенчмарк для стремления |
| Competitor C (опц.) | Сильный по онлайн-присутствию | Показать что возможно digital-first |

Выбор — по шаблону, а не хаотично. Alena подтверждает на ревью.

### 10.3. Отключаемый модуль

Competitor Benchmark — флаг `include_competitor_benchmark: true/false`.
Для некоторых B2B/нишевых клиентов конкуренты в классическом виде
не релевантны. В таком случае модуль отключается.

### 10.4. Как используется в доменных агентах
SEO, UX, Marketing агенты сравнивают не "в вакууме", а vs конкуренты:
- Где конкуренты явно сильнее/слабше.
- Структура, контент, офферы, скорость, языки.
- Что можно "позаимствовать" или обойти.

**Пример инсайта:** "У конкурента A есть DE-версия сайта, у вас — нет.
При этом 40% туристов Майорки — немецкоговорящие."

### 10.5. В отчёте: Competitor Benchmark

| Домен | Client | Competitor A | Competitor B |
|-------|--------|-------------|-------------|
| Tech Infrastructure | 3/5 🟡 | 2/5 🟠 | 4/5 🟢 |
| SEO | 2/5 🟠 | 4/5 🟢 | 3/5 🟡 |
| ... | ... | ... | ... |

### 10.6. Отраслевые baseline'ы (v2)
Накопить данные из аудитов → сформировать:
"Средний Tech Infrastructure score для отелей Майорки: 2.5/5".
Это усилит эмоциональный эффект: "вы на уровне / выше / ниже рынка".

## 12. Метрики качества пайплайна

Трекаем не только клиентские KPI, но и качество самого процесса:

| Метрика | Зачем |
|---------|-------|
| % покрытых страниц (vs лимит 30) | Насколько полный краулинг |
| % авто-детектов, помеченных Alena как "wrong" | Точность tech detection |
| Среднее время от запуска до черновика | Операционная эффективность |
| Количество рекомендаций, удалённых при ревью | Качество фильтрации агентов |
| % `[UNKNOWN]` маркировок в итоговом отчёте | Полнота данных |
| Количество `[DISCREPANCY]` между агентами | Consistency |

Цикл улучшения: после каждых 5 аудитов анализируем метрики
и корректируем промпты/паттерны агентов.

## 14. Data & Privacy

Поскольку пайплайн работает с данными клиента и активным краулингом:

- **Что сохраняется:** строки аудита в **Supabase** (`audits`, `intake_brief`, `audit_recon`, `audit_domains`, `audit_strategy`, `collected_data`, `pipeline_events`, `review_points`), экспорт отчёта по запросу (`GET /api/audits/:id/report`).
- **Где хранится:** managed PostgreSQL (Supabase) + приложение на Vercel/Railway; договор с провайдерами и RLS/Auth для изоляции пользователей.
- **Как долго:** до завершения проекта + политика ретенции (например, 1 год архив) — по договору с клиентом.
- **Анонимизация:** данные для Knowledge Base (v2) обезличиваются (без URL, имён, контактов).
- **GDPR:** клиент подписывает согласие на обработку перед аудитом.
- **Crawling:** User-Agent идентифицирован как бот GLC с контактом/политикой (см. операционные настройки краулера).

## 16. Success Criteria v1

**Go-live приоритет:** до полного масштабирования Full Audit вывести к клиентам **Free UX Snapshot** (lead magnet, предсказуемый output) и **Express Audit** (фазы 0–4, отчёт для платника). Детали приоритизации — [ARCHITECTURE-upd.md](./ARCHITECTURE-upd.md) (раздел **10**).

Как мы поймём, что v1 удался:

1. **Полный аудит ≤ 8 часов** общего времени (от URL до финального отчёта),
   включая интервью и ревью.
2. **< 15% рекомендаций** снимаются на human-ревью как неверные или нерелевантные.
3. **0 фактических ошибок** в финальном отчёте (после ревью).
4. **3+ клиента** провели аудит и дали положительную обратную связь.
5. **1+ клиент** купил Audit + Sprint (внедрение quick wins) после аудита.
6. **Free UX Snapshot** генерирует ≥ 3 лида в месяц.

## 17. Известные ограничения v1

- **Нет Lighthouse API** — performance-данные пока через web-парсинг.
- **False positives в tech detection** — regex-паттерны иногда ловят лишнее. Claude API фильтрует при интерпретации.
- **Нет доступа к внутренним системам клиента** — только через бриф и интервью.
- **Один язык отчёта** — Markdown на английском. Мультиязычные отчёты — в v2.
- **Оркестрация** — через приложение и API, не через локальный Python CLI; v2+ может добавить n8n для отдельных интеграций без замены core pipeline.
- **Нет отраслевых benchmark'ов** — пока нет базы для сравнения. Накопим после 10+ аудитов.
- **Конкурентный анализ ограничен** — только Recon-уровень по конкурентам, не полный аудит.
- **Free Snapshot — только UX** — в v2 можно добавить выбор домена.
